# 6.32 Database Indexing and Table Types - Study Notes

## Overview

Understanding database schema design including primary/secondary indexes, autoincrement behavior, and MyISAM vs InnoDB tradeoffs is essential for WordPress performance.

**Key Reference:** [Indexing - Section 4.5](../section-04/4.5-indexing.md) | [MySQL Storage Engines](https://dev.mysql.com/doc/refman/8.0/en/storage-engines.html)

---

## 1. Understanding Primary/Secondary Indexes

### Primary Index

**Characteristics:**
- Automatically created on primary key
- Unique and not null
- One per table
- Fastest lookups

**WordPress Example:**
```sql
-- wp_posts.ID is primary key
PRIMARY KEY (ID)
```

### Secondary Index

**Characteristics:**
- Created on non-primary columns
- Multiple per table
- Improves query performance
- Can be composite

**WordPress Example:**
```sql
-- Secondary index on post_type
KEY post_type (post_type)
```

---

## 2. Interpreting Autoincrement Behavior

### Autoincrement

**Behavior:**
- Automatically increments
- Unique values
- Used for primary keys
- Gaps possible (deletes)

**WordPress:**
```sql
ID bigint(20) unsigned NOT NULL AUTO_INCREMENT
```

---

## 3. Comparing MyISAM vs InnoDB Tradeoffs

### MyISAM

**Characteristics:**
- Table-level locking
- Faster for reads
- No transactions
- No foreign keys

**Use Cases:**
- Read-heavy tables
- Log tables
- Temporary data

### InnoDB

**Characteristics:**
- Row-level locking
- ACID transactions
- Foreign key support
- Crash recovery

**WordPress Default:**
- WordPress uses InnoDB
- Better for concurrent writes
- Data integrity

---

## Exam Tips

### Key Points to Remember

1. **Indexes:**
   - Primary: fastest, unique
   - Secondary: multiple, performance
   - Composite: multiple columns

2. **Autoincrement:**
   - Auto-incrementing
   - Unique values
   - Gaps possible

3. **Storage Engines:**
   - MyISAM: read-heavy
   - InnoDB: write-heavy, transactions
   - WordPress uses InnoDB
